> 这不是“功能清单”，而是一份更残酷的产品诊断。
>
> 目标不是证明 `agent-hub` 已经很强，而是回答一个更现实的问题：
>
> **为什么它现在看起来不像一个能打仗的 AI 军团，而像一堆页面、一堆概念、一堆未跑通的半成品。**

---

## 0. 一句话结论

当前 `agent-hub` 的最大问题，不是功能少，而是：

> **没有主航道，没有单一北极星，没有让用户 30 秒看懂“我该从哪里发需求、它怎么接、谁在执行、结果在哪里”的产品结构。**

所以用户看到的是：

- 页面很多
- 名词很多
- 入口很多
- Agent 很多
- 概念很多

但看不到：

- 核心页面
- 核心动作
- 核心任务流
- 核心结果
- 核心价值

这就导致它更像“Agent 功能展柜”，而不是“AI 军团产品”。

---

## 1. 现在为什么看着头疼

### 1.1 导航层级失控，用户第一眼根本不知道重点

从 `src/App.vue` 可以直接看到，左侧导航里堆了大量一等入口：

- 控制台
- 模型实验室
- Agent Stack
- Agent Console
- AI 军团流水线
- Workflow Builder
- 专家工作台
- MCP 服务器
- 评测实验室
- 计划收件箱
- 代码索引
- Agent 周报
- Agent 观测台
- 技能中心
- 一长串核心智能体
- 一长串辅助智能体

这不是“丰富”，这是**注意力打散**。

用户会立刻出现 4 个问题：

1. 我应该去哪一页开始？
2. `Agent Stack`、`Agent Console`、`AI 军团流水线` 到底谁是主页面？
3. `Workflow Builder` 和 `Pipeline` 是什么关系？
4. `Agent`、`Skills`、`MCP`、`Eval`、`Insights` 是用户要用的，还是开发者自己要看的？

结论：

> 当前 IA（信息架构）不是“产品导航”，而是“研发目录”。

---

### 1.2 首页没有把用户带入主任务，而是在分散注意力

`src/views/Dashboard.vue` 当前首页的问题非常典型：

- 上来先放多个 teaser
- 再放多个入口广告
- 再放三组 agent 卡片

看起来很热闹，但没有一个明确动作：

- 没有“发一个需求”
- 没有“查看我当前任务”
- 没有“审批计划”
- 没有“验收交付”

也就是说，首页没有在服务主任务，只是在展示“我们有这些页面”。

这会让用户产生一种非常糟糕的感觉：

> 好像哪里都能点，但不知道点了有什么结果。

---

### 1.3 `AI 军团` 这个词是对的，但产品并没有真的围绕它设计

现在的页面结构其实暴露出一个本质问题：

- 你在讲“AI 军团”
- 但产品在组织方式上，仍然是“单页功能集合”

真正的 AI 军团产品，应该围绕这几个对象来组织：

1. **任务**
2. **计划**
3. **执行**
4. **交付**
5. **角色**

而现在产品组织方式更像：

1. 页面
2. 实验室
3. 控制台
4. Builder
5. 各类概念页

这意味着：

> 你在卖“组织能力”，但 UI 呈现的是“工具箱目录”。

---

## 2. 现在为什么业务发了也跑不通

### 2.1 主链路没有被做成唯一真入口

对用户来说，唯一重要的链路应该是：

`提需求 -> 出计划 -> 审批 -> 执行 -> 交付 -> 验收`

但当前系统里，这条链路被拆散在多个地方：

- 发需求：可能在 Web、OpenClaw、IM、API
- 审批：`Plan Inbox`
- 执行：`Pipeline`
- 观测：`Insights`
- 结果：任务详情
- Agent：又有单独聊天页

这会导致一个典型问题：

> 所有环节都“有页面”，但没有一个地方是真正的“任务总控”。

---

### 2.2 `Workflow Builder` 和实际执行系统没有统一心智

从 `src/views/WorkflowBuilder.vue` 和 `backend/app/api/workflows.py` 看，
前端已经有画布、保存、加载、运行、SSE 状态展示；
但后端 workflow 仍然更像文档存储，而不是成熟执行资产。

对用户来说就会产生非常强的错位感：

- 以为这是正式工作流系统
- 实际上很多时候更像“设计稿 + 调用 pipeline 的外壳”

这会极大损伤信任：

> 画布很像真的，但执行并不让人相信它是平台核心。

---

### 2.3 多角色很多，但“谁真的在工作”不清楚

你现在展示了大量 agent。

但用户真正关心的是：

- 谁接单
- 谁拆解
- 谁设计
- 谁开发
- 谁测试
- 谁验收

如果系统不能在任务详情里清楚显示：

- 当前阶段负责人
- 上一阶段产物
- 下一阶段等待条件
- 为什么卡住

那“14 个角色”只会像营销词，而不是可见的协作结构。

结论：

> 当前更像“列出了很多 Agent”，而不是“让用户真正看见团队在协作”。

---

### 2.4 失败态不够直白，导致“看起来没效果”

当业务跑不下去时，用户最需要的是：

- 卡在哪一关
- 卡住原因
- 谁需要处理
- 下一步怎么办

现在很多页面更偏“系统视角”而不是“业务视角”：

- event log
- trace
- spans
- token
- observability

这些对研发有帮助，但对业务侧没有形成“下一步指引”。

于是用户的感受就会变成：

> 页面很多，数据很多，日志很多，但任务为什么没完成，还是不知道。

---

## 3. 现在为什么模型接入看起来非常混乱

### 3.1 模型入口太多，角色定位不清

当前至少存在这些模型相关概念：

- `llm_router`
- OpenAI 兼容代理
- provider fallback
- settings 里的模型配置
- model lab
- pipeline 中不同阶段模型
- OpenClaw / 外部平台桥接

这些能力本身并不一定错，但产品层没有给出清晰模型：

用户会问：

1. 我到底是在配置“平台默认模型”还是“单任务模型”？
2. `Model Lab` 是实验页还是正式配置页？
3. `openai_compat` 是给第三方平台接的，还是给自己内部调用的？
4. 失败 fallback 到底是透明的还是可见的？

如果用户答不上来，就会觉得“模型接入非常乱”。

---

### 3.2 技术上在兜底，产品上却没讲清楚

`backend/app/services/llm_router.py` 做了很多兼容和 fallback。

这在工程上是加分的，但在产品上也埋下一个问题：

- 平台到底支持哪些模型
- 谁是主模型
- 谁是便宜模型
- 谁负责 planning
- 谁负责 coding
- 谁负责 review

如果没有一个统一“模型策略视图”，用户只会觉得：

> 好像哪里都能配模型，但不知道最终到底是谁在干活。

---

## 4. 为什么现在更像 A-agent，而不像 AI-Agent

这是你这次吐槽里最准确的一句。

### “A-agent”的典型特征

- 有 Agent 页面
- 有 Agent 名字
- 有 Agent prompt
- 有很多概念
- 有聊天入口

但没有：

- 真任务闭环
- 真协作结构
- 真执行编排
- 真质量控制
- 真交付结果

也就是：

> 像是在“展示 Agent”，不是在“让 Agent 完成工作”。

---

### 真正的 AI-Agent 产品应该让用户看到什么

不是“我们有多少个 Agent”，而是：

- 这个需求已经被谁接收
- 当前计划是什么
- 哪个角色在执行哪一步
- 哪一步失败了，为什么
- 最终交付物在哪里
- 用户如何批准、打回、继续

也就是说，真正的主角不是 `Agent 卡片`，
而是 `任务的推进`。

结论：

> 当前 UI 把“Agent 本身”放在中心，导致“任务结果”反而退到了二线。

---

## 5. 这不是加页面能解决的，而是要删页面、并页面、改主线

### 5.1 必须先定唯一北极星

`agent-hub` 只能有一个一眼能懂的产品定义：

> **面向需求交付的 AI 军团系统**

不是：

- AI 平台
- 多模型平台
- workflow 平台
- Agent 实验室
- builder 集合

这些都只能是子能力。

---

### 5.2 页面结构必须从“功能目录”改成“任务主线”

建议只保留 4 个一级页面：

1. **任务**
   - 新建需求
   - 任务列表
   - 任务详情
   - 审批 / 验收

2. **军团**
   - 当前角色体系
   - 模板化执行编队
   - 谁负责什么

3. **能力**
   - 模型策略
   - 技能
   - MCP / 工具
   - 知识库

4. **平台**
   - 集成
   - 观测
   - 设置

然后把下面这些二级化或隐藏：

- `Agent Stack`
- `Agent Console`
- `Model Lab`
- `Eval Lab`
- `Insights`
- `Codebase Lab`
- `MCP Servers`

它们应该服务平台，不应该和主任务抢第一层注意力。

---

### 5.3 首页必须只做一件事：发需求并看任务推进

建议首页重做成：

#### 顶部
- 一个大输入框：`告诉 AI 军团你要做什么`
- 两个选项：
  - `先给方案`
  - `直接执行`

#### 中部
- 我的待办：
  - 待审批
  - 执行中
  - 待验收

#### 底部
- 最近任务卡片
- 每张卡只显示：
  - 标题
  - 当前阶段
  - 当前负责人
  - 风险 / 卡点

不再展示一堆 teaser。

---

### 5.4 `Agent` 页面不能再是主入口，只能是任务的展开视图

现在太像“点一个 Agent 去聊天”。

但真正的 AI 军团不应该鼓励用户先选 Agent，
而应该鼓励用户先发任务。

建议：

- 用户默认不直接进入 `AgentChat`
- 而是先创建任务
- 任务内部再展示当前角色对话 / 阶段协作

这样用户感知到的是：

> 我在跟一支军团打交道，不是在一个个点顾问。

---

## 6. 迭代方向：从“展示很多”改成“完成一条”

下面是建议的迭代方向，不是功能扩张，而是产品收敛。

### Phase 1：收敛产品叙事

目标：

- 任何用户 30 秒看懂产品

动作：

- 重写首页
- 重构一级导航
- 删除 teaser 式首页结构
- 隐藏 Agent 系列页面到次级入口

---

### Phase 2：打通唯一任务主链路

目标：

- 让一条任务从需求到验收全程可见

动作：

- 统一任务入口
- 强化 `Plan -> Approve -> Execute -> Deliver -> Accept`
- 所有状态都回到一个任务详情页里
- 任务详情页显示：
  - 当前阶段
  - 当前角色
  - 当前阻塞
  - 当前输出

---

### Phase 3：把“军团”可视化，而不是把“Agent 列表”可视化

目标：

- 让用户真正看见协作

动作：

- 每个任务显示角色泳道
- 角色间的交接、返工、审批可见
- 显示“是谁把任务打回了”
- 显示“为什么卡住”

---

### Phase 4：模型与能力统一收口

目标：

- 模型接入不再让人迷惑

动作：

- 建立唯一的“模型策略页”
- 分清三类模型：
  - 默认模型
  - 阶段模型
  - 回退模型
- 对用户隐藏不必要的实现细节
- `Model Lab` 变成次级工具，而不是主入口

---

### Phase 5：把平台能力藏到后面，把结果放到前面

目标：

- 用户只看结果和进度，平台能力按需展开

动作：

- `MCP`、`Eval`、`Codebase`、`Observability` 全部二级化
- 普通用户只看任务
- 高级用户再进入能力和平台层

---

## 7. 产品上的真正取舍

现在最重要的，不是继续加：

- 更多 agent
- 更多页面
- 更多实验室
- 更多看起来高级的概念

而是要敢于砍掉这些错觉：

- “功能多就说明强”
- “页面多就说明完整”
- “Agent 多就像军团”
- “日志多就说明可观测”

真正需要的是：

> **一个用户能稳定发需求、稳定看到执行、稳定拿到交付的主流程。**

这才是产品。

---

## 8. 最后的结论

你现在的判断是对的：

> 当前的 `agent-hub` 还没有把“AI 军团”做成产品。

它现在更像：

- 一个有不少潜力的后端系统
- 一个叠了很多概念的前端
- 一个把内部模块直接暴露给用户的控制台

它缺的不是再加 20 个功能，
而是一次彻底的产品收敛：

- **删入口**
- **并页面**
- **定主线**
- **把任务放中间**
- **把 Agent 放到任务里**
- **把平台能力放到后台**

只有这样，用户才会真正感觉到：

> 这不是一堆 A-agent。
>
> 这是一个能接需求、能出方案、能推进执行、能交付结果的 AI-Agent 军团。
